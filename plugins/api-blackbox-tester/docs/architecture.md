# 架构说明

`api-blackbox-tester` 是一个由多个独立 Skill 组成的 Codex 插件。各 Skill 按测试生命周期拆分，使测试方案、执行验证和最终报告可以独立审查、重复执行，并保持清晰的证据链。

## 组件关系

```text
api-blackbox-tester
        |
        | 根据用户意图选择阶段
        v
+-----------------------------+
| api-blackbox-test-planner   |
+-----------------------------+
        |
        | 测试方案、用例 ID、数据计划、编排链路
        v
+-----------------------------+
| api-blackbox-test-executor  |
+-----------------------------+
        |
        | curl/grpcurl 证据、DB 校验、失败原因
        v
+-----------------------------+
| api-blackbox-test-reporter  |
+-----------------------------+
```

## Skill 职责

### `api-blackbox-tester`

统一编排入口。它判断用户需要测试方案、执行验证、测试报告，还是完整的阶段化流程。

该 Skill 不承载具体测试细节，只负责阶段选择、流程串联和交付物衔接要求。

### `api-blackbox-test-planner`

测试方案阶段。它在可用时读取需求、项目上下文、路由定义、DTO、中间件、已有测试、API 文档、配置和数据库访问逻辑。

它输出以下内容：

- 测试范围和假设。
- 兼容性验证。
- 测试数据计划。
- 功能测试用例。
- 边界测试。
- 安全测试。
- 集成或接口编排链路。
- 补充验证项。
- 性能测试建议。

### `api-blackbox-test-executor`

执行验证阶段。它以真实接口行为为准，并在需要数据库准备或校验时优先使用 `db-mcp`。

它记录以下证据：

- 执行环境。
- 数据准备。
- 请求命令。
- 响应结果。
- 数据库验证。
- 编排链路步骤。
- 清理状态。

### `api-blackbox-test-reporter`

报告阶段。它汇总 planner 输出和 executor 证据，评估：

- 需求覆盖率。
- 用例执行覆盖率。
- 接口覆盖率。
- 边界测试覆盖率。
- 安全测试覆盖率。
- 集成链路覆盖率。
- 数据场景覆盖率。
- 兼容性覆盖率。
- 性能测试建议覆盖率。

缺少真实执行证据时，reporter 不得将变更标记为通过。

## 阶段交付契约

阶段之间依赖稳定 ID 和明确证据：

| 生产方 | 输出 | 消费方 |
| --- | --- | --- |
| Planner | `tests/【需求】_YYYYMMDD/测试方案.md`，包含用例 ID、链路 ID、数据计划、预期响应和 DB 状态 | Executor |
| Executor | `tests/【需求】_YYYYMMDD/执行记录.md`，包含 curl/grpcurl 命令、响应、DB 校验、失败原因 | Reporter |
| Reporter | `tests/【需求】_YYYYMMDD/测试报告.md`，包含覆盖率、测试结论、剩余风险、发布建议 | 用户或发布流程 |

三个阶段必须复用同一需求目录。`【需求】` 取用户需求的简短标题，`YYYYMMDD` 取执行当天日期或用户指定日期。

## 外部依赖

`db-mcp` 对规划阶段不是必需项，但在执行阶段需要数据库准备或校验时应使用。它可用于查看表结构、创建或复用测试数据、验证副作用，以及清理测试归属的数据。

HTTP 接口执行通常使用 `curl`。gRPC 接口执行通常使用 `grpcurl`。

## 设计约束

- 每个 Skill 聚焦单一阶段。
- 下游阶段可消费的输出格式必须保持清晰。
- 三个阶段都必须写入对应 Markdown 文件，不能只在对话中输出。
- 结论以证据为准，不以推测为准。
- 环境、凭证或依赖缺失时，应判定为阻塞，而不是通过。
- 未明确授权时，不执行破坏性操作。
