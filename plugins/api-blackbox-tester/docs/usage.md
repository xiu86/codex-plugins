# 使用指南

本文说明如何在接口开发和验证过程中使用 `api-blackbox-tester` 插件。

## 选择 Skill

如果希望 Codex 自动判断当前需要哪个阶段，使用统一入口：

```text
使用 $api-blackbox-tester 编排接口黑盒测试的方案、执行和报告。
```

如果已经明确需要某个阶段，可以直接使用对应 Skill：

| 需求 | Skill |
| --- | --- |
| 测试范围、用例、边界测试、安全测试、接口编排方案 | `$api-blackbox-test-planner` |
| 真实 HTTP/gRPC 执行、curl 证据、数据库校验 | `$api-blackbox-test-executor` |
| 覆盖率、测试结论、失败汇总、发布建议 | `$api-blackbox-test-reporter` |

## 阶段一：测试方案

示例提示词：

```text
使用 $api-blackbox-test-planner 根据这个需求和当前项目代码输出接口黑盒测试方案：
<粘贴需求或变更说明>
```

planner 应输出：

- 测试范围和假设。
- 兼容性验证。
- 测试数据计划。
- 正常和异常测试用例。
- 边界测试。
- 安全测试。
- 多接口编排或集成链路。
- 补充验证项。
- 性能测试评估与建议。

本阶段必须写入：

```text
tests/【需求】_YYYYMMDD/测试方案.md
```

## 阶段二：执行验证

示例提示词：

```text
使用 $api-blackbox-test-executor 按下面测试方案执行真实接口请求。
base URL: http://127.0.0.1:10008
鉴权方式: Bearer token，token 已在环境中准备
测试方案:
<粘贴 planner 输出>
```

executor 需要以下信息：

- 安全的测试环境。
- base URL 或 gRPC endpoint。
- 必需的 token 或凭证。
- 需要数据库准备或校验时的数据库实例信息。
- 创建或复用测试数据的授权。

executor 应记录：

- 测试环境和基础配置。
- 测试数据准备记录。
- 可复现的 curl 或 grpcurl 命令。
- 实际响应状态码、响应头和响应体摘要。
- 数据库验证证据。
- 编排链路步骤、提取字段和传递值。
- 失败、阻塞、跳过和清理状态。

本阶段必须写入：

```text
tests/【需求】_YYYYMMDD/执行记录.md
```

## 阶段三：测试报告

示例提示词：

```text
使用 $api-blackbox-test-reporter 根据下面测试方案和执行证据输出最终报告：

测试方案:
<粘贴 planner 输出>

执行证据:
<粘贴 executor 输出>
```

reporter 必须基于真实执行证据给出最终结论。缺少实际请求和必要数据库验证时，只能给出阻塞或证据不足结论，不能标记为通过。

本阶段必须写入：

```text
tests/【需求】_YYYYMMDD/测试报告.md
```

## 文件输出规范

三个阶段默认在项目根目录下使用同一个需求目录：

```text
tests/【需求】_YYYYMMDD/
```

目录和文件命名规则：

- `【需求】` 使用用户需求的简短标题；未提供标题时由 Skill 根据需求内容提炼 8 到 20 个中文字符。
- `YYYYMMDD` 使用执行当天日期，例如 `20260428`；如果用户明确指定日期，以用户指定日期为准。
- 文件名中的 `/`、空格、冒号、引号等不适合作为路径的字符应替换为 `_`。
- executor 和 reporter 应优先复用前序阶段已经创建的目录。
- 每个阶段完成后，回复中必须给出对应文件路径。

## 证据记录建议

- 每个已执行请求都保留可复现的 curl 或 grpcurl 命令。
- 脱敏 secret 值，但保留 header 名称和鉴权方式。
- 写接口同时验证响应和数据库状态。
- 参数校验失败的用例需要验证没有产生意外数据库写入。
- 编排链路记录每一步，而不是只记录最终结果。
- 环境、凭证、数据或依赖缺失时，明确标记为阻塞。

## 安全使用说明

- 破坏性或侵入式 payload 应在隔离测试环境执行。
- 生成的测试记录建议使用 `codex_api_test_` 等可识别前缀。
- 只清理明确由测试创建或归属测试的数据。
- 未经授权不得对生产系统或第三方系统执行安全测试。
