# 接口黑盒测试报告模板

执行接口测试后使用本结构输出报告。

## 概览

- 测试目标：方法、路径、环境、构建版本或提交。
- 最终结论：通过、失败、阻塞或部分通过。
- 执行范围：通过、失败、跳过、阻塞的用例数量。
- 兼容性结论：通过、失败、不适用或证据不足。
- 覆盖率结论：充分、基本充分、不足或阻塞。

## 编码前测试方案摘要

| 类别 | 摘要 | 备注 |
| --- | --- | --- |
| 测试范围 | 范围内/范围外 | 说明假设 |
| 兼容性验证 | 计划验证的契约点 | 说明风险 |
| 测试数据 | 计划准备的数据 | 说明构造或获取方式 |
| 测试用例 | 计划用例数量和重点场景 | 说明优先级 |
| 边界测试 | 计划边界参数和值 | 说明风险 |
| 安全测试 | 计划安全场景和 payload | 说明隔离要求 |
| 补充验证项 | 需求未列举但应验证的场景 | 说明来源 |
| 接口编排/集成测试 | 计划链路、步骤和依赖 | 说明是否包含非本次变更接口、数据传递和分支 |
| 性能测试建议 | 建议性能测试的接口或链路 | 说明原因、场景和指标 |

## 接口请求记录

所有已执行接口请求都要记录为 curl 形式。

| 用例 ID | curl 请求 | 请求参数 | 返回数据 | 是否通过 | 不通过原因 |
| --- | --- | --- | --- | --- | --- |
| API-001 | `curl -sS -X POST ...` | header/query/body 摘要 | 状态码和响应体摘要 | 通过/失败 | 失败或阻塞原因 |

## 接口编排/集成链路

复杂需求涉及多接口依赖时，记录链路和每一步的数据传递。链路可以包含本次未变更但业务流程必需的接口。

| 链路 ID | 场景 | 步骤 | 是否本次变更接口 | 依赖关系 | 提取数据 | 传递到 | 步骤结果 |
| --- | --- | --- | --- | --- | --- | --- | --- |
| FLOW-001 | 创建后查询 | 1. 创建 2. 查询 | 是/否 | 第 2 步依赖第 1 步返回 ID | `$.data.id` | 查询接口 path/body | 通过/失败 |

## 失败项

每个失败项说明：

- 用例 ID。
- 可复现请求。
- 实际响应和状态码。
- 预期行为。
- 数据库证据。
- 有证据支持时，写出疑似原因。

## 兼容性说明

记录以下契约是否变化：

- HTTP 状态码。
- 响应结构和必需字段。
- 错误码和错误消息契约。
- 分页、排序、筛选、幂等和鉴权行为。
- 边界参数处理和非法输入响应。
- XSS、注入、越权、敏感信息泄露等安全风险防护。
- 多接口链路中的字段传递、状态流转、步骤顺序、非变更接口依赖和失败处理语义。
- 关键接口或链路的性能特征，例如响应时间、分页默认值、返回数据量、缓存命中路径或依赖调用次数。
- 现有客户端可感知的数据库副作用。
